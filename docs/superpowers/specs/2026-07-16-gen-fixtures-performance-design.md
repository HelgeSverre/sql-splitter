# Make `gen-fixtures` faster

`gen-fixtures` can generate SQL faster by removing temporary strings from its hot paths. The work will proceed in measured stages: eliminate allocation-heavy formatting first, replace multi-tenant row batches only if profiling still supports it, and change random-string generation only if that produces a substantial additional gain.

## Goals

- Reduce release-build wall time for both simple `--rows` and multi-tenant `--scale` generation.
- Keep memory bounded and preserve the existing streaming design.
- Keep two runs with the same arguments and seed byte-for-byte identical.
- Preserve the current generated bytes through the allocation-removal stages.
- Let measurements decide whether a later random-generation change justifies changing seeded output across versions.

This work will not parallelize generation, change SQL syntax, add CLI options, or optimize the non-streaming `Generator` and `Renderer` APIs.

## Baseline

Measurements were taken on an Apple M2 Max with the release profile. Hyperfine used two warmups and ten measured runs; output was written to `/dev/null` to isolate generation cost.

| Workload                                    |       Output size |    Mean wall time | Peak RSS from file-output run |
| ------------------------------------------- | ----------------: | ----------------: | ----------------------------: |
| Simple MySQL, 10 tables × 125,000 rows      | 129,722,365 bytes | 1.039 s ± 0.020 s |                      3,152 KB |
| Simple PostgreSQL, 10 tables × 125,000 rows | 115,076,690 bytes | 0.722 s ± 0.005 s |                      2,528 KB |
| Multi-tenant MySQL, `xlarge`                | 134,491,966 bytes | 1.574 s ± 0.014 s |                     31,536 KB |
| Multi-tenant PostgreSQL, `xlarge`           | 137,024,950 bytes | 1.276 s ± 0.042 s |                     30,656 KB |

Stages 1 and 2 must retain these SHA-256 hashes for seed `12345` and the default batch size:

| ------------------------------------------- | ------------------------------------------------------------------ |
| Simple MySQL, 10 tables × 125,000 rows      | `4bdaa3e6f0b7d23bd7eb13277a37f31dc5712bbf4f91980eb5b44dd48ed8a032` |
| Simple PostgreSQL, 10 tables × 125,000 rows | `ff5aaf7c3c639a288bec707f660e7d51af7f581865a017ee8856b439f8d15046` |

Increasing `--batch-size` from 100 to 1,000 or 10,000 changed simple-mode time by about 1% and did not improve multi-tenant time. Batch size is not an optimization target.

Symbols-enabled Samply profiles identified these inclusive costs:

- Simple MySQL: `format_string` accounted for 44.6% of samples and random-string construction accounted for 42.7%. ChaCha generation and `random_range` together accounted for 35.3% of self samples.
- Multi-tenant MySQL: row `format!` construction accounted for 49.7% of samples and `format_string` accounted for 43.8%.
- Allocation, freeing, `memmove`, and repeated `str::replace` calls dominated self time in both modes.

The inclusive percentages overlap because formatting calls contain allocation and copying work.

## Stage 1: Remove temporary SQL strings

Introduce a private borrowed value that implements `Display` for a quoted SQL string. It stores a `Dialect` and `&str`, then writes the opening quote, escaped content, and closing quote directly into the caller's formatter.

The formatter must reproduce the current escaping exactly:

| Dialect    | Prefix | Escapes                                                                    |
| ---------- | ------ | -------------------------------------------------------------------------- |
| MySQL      | `'`    | `\` → `\\`, `'` → `\'`, newline → `\n`, carriage return → `\r`, tab → `\t` |
| PostgreSQL | `'`    | `'` → `''`                                                                 |
| SQLite     | `'`    | `'` → `''`                                                                 |
| MSSQL      | `N'`   | `'` → `''`                                                                 |

Both streaming generators will return this borrowed formatter from `format_string` instead of allocating an escaped `String` and then another quoted `String`. Existing `format!`, `write!`, and `writeln!` call sites can consume the formatter without changing their SQL layout.

Simple generation will also replace `random_string(len) -> String` with a buffer-filling method. Each batch will allocate name and description buffers once, clear them for each row, and refill them using the existing `ChaCha8Rng` calls in the same order. PostgreSQL COPY values can use the buffers directly because the generated alphabet contains no COPY escape characters.

This stage must preserve the full output hashes recorded in the baseline.

## Stage 2: Consolidate multi-tenant row batches

Run the four representative benchmarks and capture a new profile after Stage 1. Proceed only if row construction, allocation, and joining remain the dominant multi-tenant cost.

If justified, replace each `Vec<String>` row batch with a private `RowBatch` containing one reusable `String` and a row count. Appending a row will:

1. Add `,\n` before every row except the first.
2. Format the row directly into the batch buffer.
3. Increment the row count.

Flushing will write the existing `INSERT` header, the buffered rows, and `;\n`, then clear the buffer without releasing its capacity. Writing to a `String` is infallible, so the helper may use `expect("writing to a String cannot fail")` to document that invariant.

The buffer will reserve a conservative amount derived from `batch_size`; it must not grow with the complete fixture. This stage must also preserve the baseline output hashes.

## Stage 3: Consider faster random-string sampling

Capture another simple-mode profile after allocation removal. Proceed only if random character selection remains a leading cost and a prototype improves simple MySQL wall time by at least 20% relative to the completed allocation stages without regressing either multi-tenant workload by more than 5%.

The candidate algorithm will keep `ChaCha8Rng` but draw bytes in blocks. Because the alphabet contains 63 characters, each byte can use its low six bits; value 63 is rejected to avoid modulo bias. This changes the byte stream produced by existing seeds, but repeated runs with the new implementation must remain identical.

Stage 3 will be a separate change so it can be rejected without discarding the output-compatible improvements.

## Correctness and performance checks

Tests will cover the formatter's exact escaping for all four dialects, deterministic generation for identical seeds, different output for different seeds, and representative simple and multi-tenant output. New helpers will be introduced test-first.

Before and after each stage:

1. Build `gen-fixtures` with the release profile.
2. Generate the baseline files and compare byte counts and SHA-256 hashes when output compatibility is required.
3. Run the four Hyperfine workloads with two warmups and ten measured runs.
4. Record peak RSS with GNU time for simple MySQL and multi-tenant MySQL.
5. Capture a symbols-enabled Samply profile for the workload targeted by the next stage.

An allocation-removal stage is successful when at least one representative workload improves by 10% without a statistically clear regression above 5% in another workload. Peak RSS must not increase by more than 10% or 1 MB, whichever allowance is larger.

Final verification runs `cargo fmt`, `cargo clippy --all-targets -- -D warnings`, the `test_data_gen` package tests, and the full workspace test suite.
